# Протокол Dev Review

> Описывает процедуру проверки постановки задачи до начала реализации.
> Введён после серии v29–v32 из-за мискоммуникации между продактом и разработчиком.
> Последнее обновление: апрель 2026.

---

## Зачем

Даша видит задачу со стороны продукта — что должно измениться в поведении.
Cursor видит задачу со стороны кода — как это затрагивает форматы данных, downstream-агентов, стоимость реализации.

Эти два взгляда раньше встречались только в момент реализации. Dev Review — точка, где они встречаются до того как что-то сделано.

---

## Статусный флоу

```
new
  → dev-review       (Cursor прочитал спек, пишет флаги)
  → spec-review      (Claude читает флаги, отвечает)
  → spec-approved    (Даша даёт go — 30 сек скан)
     или
  → blocked-on-product (есть [PRODUCT] флаги без ответа от Даши)
  → in-progress      (Cursor реализует)
  → dasha-review     (Даша проверяет результат)
  → verified-on-run  (фикс прогнан на пилоте с зелёным выхлопом)
  → done
```

**Правила:**

1. Задача не переходит в `in-progress` без `spec-approved`.
2. Задача не переходит в `done` без `verified-on-run` (см. ниже).

---

## Два класса провалов и две защиты от них

Введено 2026-04-30 после разбора Claude Opus, разделено на artifact-провалы и holistic-провалы.

### Artifact-провал

**Симптом:** все промежуточные метрики PASS, но финальный продукт сломан.

**Пример:** регрессия 023. validate_layout_fidelity сказал PASS на 58 абзацев, Cursor отчитал «35 страниц», PDF получил 117KB и exit code 0. Никто не открыл PDF — он содержал 1059 символов вместо 13000+, 92% контента молча отсутствовало.

**Защита:** `verified-on-run` (см. ниже). Открыть финальный артефакт, не верить промежуточным сигналам.

### Holistic-провал

**Симптом:** каждая задача чиста в изоляции, дыры появляются на стыках.

**Пример:** 3 находки Opus после 016/017/018. validate_fidelity вызывался, но результат не использовался. photos_dir guard в одной ветке, но не в симметричной. Auto-patch не учитывал hybrid-элементы. Каждое — корректная реализация спеки своей задачи. Дыра в том что ни одна спека не описывала **взаимодействие** между задачами.

**Защита:** `audit-checkpoint` + регулярный external second opinion (см. ниже). Чтение кода без задачи в голове, поиск противоречий между задачами.

**Не путать.** Это разные ментальные операции. verified-on-run на artifact-провале не поймает holistic-провал и наоборот. Если работать одной защитой — половину времени будешь не в том режиме.

## Защита 1: статус `verified-on-run`

**Правило:** ни одна системная задача (правка кода / промпта / архитектуры) не закрывается как `done`, пока её фикс не прогнан end-to-end и **финальный артефакт не открыт**.

### Posture-forcing требование при закрытии

При переходе из `dasha-review` в `verified-on-run` тот, кто закрывает, **обязан написать одно предложение конкретного наблюдения о финальном артефакте.** Не метрики, не PASS/FAIL, не «файлы на местах» — **что именно видно при открытии**.

**Примеры правильного:**
- «Открыл PDF v38, на стр 5 подглава "Учёба и первая работа" с текстом "В Старобельске Валентина...", всего 17 страниц 14 726 символов»
- «fact_map_v37.json: person_004 Полина Амельченко aliases=['тётя Поля'], без 'Рудая Пелагея Алексеевна' и 'Марфа'»
- «Прогнал normalize с битым layout без --allow-mismatch — получил exit code 1, в stdout три ошибки fidelity»

**Примеры НЕдопустимого:**
- ❌ «validate_fidelity PASS, файлы на местах»
- ❌ «Cursor отчитался об успехе»
- ❌ «Прогон завершён без ошибок, exit code 0»
- ❌ «Статическая верификация Claude пройдена»

**Категория «статическая верификация» из протокола удалена** — это согласие с автором, а не верификация. Если вместо предложения наблюдения о реальном артефакте получается «прочитал диф и согласен» — задача не закрыта, возврат в `in-progress`.

### Что считается «зелёным выхлопом»

- **Для код-задачи:** прогон затронутого этапа пайплайна на пилоте. Финальный артефакт открыт, написано предложение наблюдения.
- **Для алгоритмического фикса** (без LLM): unit-test или repro-скрипт запущен на конкретном входе, выход проверен глазами или сравнением с эталоном.
- **Для промпт-патча:** прогон агента, ответ открыт, проверено что новое правило применено хотя бы раз.

### Кто отвечает

- **Cursor:** после реализации — запустить, приложить артефакты, написать предложение наблюдения.
- **Claude:** не закрывает в `verified-on-run` без предложения наблюдения. Сама пишет своё предложение наблюдения о том же артефакте — если Cursor пишет «PDF на месте», Claude открывает PDF и пишет «вижу 17 страниц, на стр 9 Глава 03». Два независимых наблюдения снижают шанс что оба смотрят не туда.
- **Даша:** financial gate — каждый прогон $2-3, утверждает «гонять или дешевле собрать batch фиксов и прогнать вместе».

### Batch verification

Несколько мелких задач можно верифицировать одним прогоном если их фиксы независимы. В этом случае каждая задача получает своё предложение наблюдения о том же прогоне (разные ракурсы артефакта).

### Почему

Регрессия 023 (2026-04-30): задача 017 закрыта как `done` после фиксов в коде. PDF v37 содержал 1059 символов из 13000+, 92% контента молча отсутствовало. Все промежуточные метрики (35 страниц / 58 абзацев / fidelity PASS / exit code 0) сообщали об успехе. Никто из четырёх (Cursor, Claude, Opus, Даша) не открыл PDF.

Posture-форсинг через предложение наблюдения — лекарство от этого. Если бы любой из четырёх обязан был написать «открыл PDF, на странице X видно Y», то один взгляд сразу бы показал что страниц мало и контент пуст. Задача не закрыта = задача не закрыта.

## Защита 2: `audit-checkpoint` (holistic-режим)

**Правило:** периодический systemic review кода без привязки к конкретной задаче.

### Триггеры (структурные, не календарные)

Аудит обязателен когда сработал любой:

1. **Кластер 2+ задач трогает один subsystem.** Пример: 016/017/018 — все про Stage 4. Каждая чистая в изоляции, дыры на стыках появились в кластере.
2. **Контракт между компонентами меняется.** Пример: после 017 формат layout.json новый — все его читатели должны быть пересмотрены, не только Layout Designer.
3. **В описании задачи слова «orchestration» / «обвязка» / «protocol».** По определению означают межфайловые взаимодействия которые в одном файле не видны.
4. **Задача планирует закрываться только статической проверкой кода.** Это запах. Не валидация — запах.
5. **Календарный backup:** прошло > 3 недель без holistic-прохода → независимо от структурных триггеров, аудит. Страховка от случая когда триггеры пропустили.

### Дисциплина прохождения

1. **Открывать код слепо, БЕЗ спеки задачи.** Сначала чтение, потом сверка с ожиданиями. Если порядок обратный — ищется подтверждение априорных ожиданий, а не противоречия.
2. **Вопрос на проходе:** «Что меня здесь удивило?» — не «Соответствует ли спеке?»
3. **Искать на стыках.** Где результат вызова не используется. Где guard есть в одной ветке но не в симметричной. Где формат на выходе одного компонента не учитывается на входе другого.
4. **Список наблюдений → новые задачи или комментарии в существующие.** Audit-проход не правит код, а размечает проблемы для отдельных задач.

### Кто и когда

- **Claude (я):** провожу аудит при срабатывании структурных триггеров. По оценке Opus — закрываю ~70% holistic-провалов сама.
- **External second opinion** (Opus или другая модель): **раз в 3-4 системные задачи** — независимый проход «слепо посмотри». Это не часть основного процесса, а **контр-якорь от блайнд-спота «не вижу чего нет»**. Дешевле и надёжнее чем растить self-audit-дисциплину до уровня поиска отсутствующих вещей.

### Почему 70/30

Self-audit структурно не закрывает блайнд-спот «не видишь чего нет». Если в моём контексте есть объяснение почему `validate_fidelity` вызывается — я не задам вопрос «а используется ли результат», потому что объяснение это скрывает. Внешняя модель без этого контекста спросит первой. Это не вопрос внимательности, это структурное ограничение позиции автора.

### Почему

Разбор Opus 2026-04-30 показал что наши 4 актора (Cursor, Claude, Opus до этого, Даша) работали в режиме incremental delivery. Каждая спека узкая, каждое ревью узкое. Holistic-режим (читать код без задачи, искать противоречия) ни в чьей роли не было. Opus за 30 минут нашёл 3 критичных бага которые мы делали 3 итерации.

## Классификация флагов

### [TECH] — технические флаги

Cursor поднимает, Claude отвечает автономно.

Примеры:
- изменение формата данных (ломает downstream-агента)
- side effect на другой стейдж пайплайна
- существует более простая альтернатива реализации
- спек неполный: не описан edge case или поведение при ошибке
- temperature / max_tokens противоречат текущим стандартам

### [PRODUCT] — продуктовые флаги

Cursor поднимает, **обязательно отвечает Даша** (не только Claude).

Примеры:
- неясно какой UX-результат считается успехом
- спек противоречит предыдущему решению по продукту
- критерий приёмки в спеке отсутствует или субъективен
- затрагивается политика (оплата, данные пользователей, роли)

---

## Лимит итераций

**1 раунд** — Cursor пишет флаги, Claude/Даша отвечают.

Если после одного обмена не сошлись → статус `blocked-on-product` → эскалация к живым людям.

Это защита от «вежливого согласия между AI» — когда модель соглашается чтобы закрыть тему, а не потому что проблема реально решена.

---

## Финальный sign-off

Даже если Dev Review прошёл быстро и без разногласий — **Даша явно даёт go**.

Это не бюрократия, а точка ответственности: после `spec-approved` Cursor начинает реализацию, и откат дороже.

Формат sign-off: Даша меняет статус задачи на `spec-approved` в файле задачи (или пишет "go" в Dev Review Response).

## Что Cursor проверяет при Dev Review

Обязательный чеклист при заполнении секции Dev Review:

- [ ] Все выходные форматы задачи совпадают с тем что ожидает следующий агент
- [ ] Нет неявного изменения формата `book_draft.json`, `fact_map.json`, `layout.json`
- [ ] Если меняется промпт — указан агент, версия и температура
- [ ] Если меняется код — описан затронутый модуль и тип изменения
- [ ] Критерий приёмки ("как проверить") конкретный и верифицируемый
- [ ] Спек не противоречит `pipeline-run-protocol.md`

---

## Исключения

Dev Review **не обязателен** для:
- задач типа `промпт` без изменения формата данных (минорная правка стиля, тона, примеров)
- задач со статусом `hotfix` (срочное исправление поломки в продакшне)

В этих случаях в поле Dev Review пишется: `skipped — [причина]`.
