## Общая схема прогона

```
┌─────────────────────────────────────────────────────────────┐
│ Pre-flight (проверка промптов и конфига)                    │
└────────────────────┬────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────┐
│ Stage 1: Cleaner → Fact Extractor                           │
│ Вход: TR1                                                    │
│ Выход: fact_map.json                                         │
└────────────────────┬────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────┐
│ Stage 2 (Phase A): Историк → Ghostwriter → Fact Checker     │
│ Вход: fact_map                                               │
│ Выход: book_draft.json                                       │
│ Цикл возврата: FC FAIL → GW revision (макс 3 итерации)      │
└────────────────────┬────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────┐
│ Stage 3: Literary Editor → Proofreader (per-chapter)        │
│ Вход: book_draft                                             │
│ Выход: book_FINAL_stage3.json                                │
└────────────────────┬────────────────────────────────────────┘
                     ↓
            ┌────────┴────────┐
            ↓                 ↓
     ┌───────────┐     ┌─────────────┐
     │ Gate 1    │     │ Phase B     │
     │ Текст     │     │ (если есть  │
     │ (.txt)    │     │ TR2)        │
     └───────────┘     └──────┬──────┘
                              ↓
                     ┌─────────────────┐
                     │ Gate 1          │
                     │ (итоговый текст)│
                     └────────┬────────┘
                              ↓
               [Ворота 2a → 2b → 2c → 3 → 4]
```

**Ключевой принцип:** каждый этап создаёт артефакт, который становится входом следующего. При сбое любого агента — **останавливаем прогон**, не идём дальше.

## Pre-flight — проверка перед стартом

**Перед каждым прогоном** разработчик обязан проверить:

### 1. Версии промптов в `pipeline_config.json`

Должны соответствовать актуальной таблице (см. `pipeline-state.md`):

| Агент | Файл промпта | Ожидаемая версия |
|-------|-------------|------------------|
| Cleaner (01) | `01_cleaner_v1.md` | v1 |
| Fact Extractor (02) | `02_fact_extractor_v3.3.md` | v3.3 |
| Историк (12) | `12_historian_v3.md` | v3 |
| Ghostwriter (03) | `03_ghostwriter_v2.6.md` | **v2.11** (внутри файла) |
| Fact Checker (04) | `04_fact_checker_v2.md` | **v2.4** (внутри файла) |
| Literary Editor (05) | `05_literary_editor_v3.md` | v3 |
| Proofreader (06) | `06_proofreader_v1.md` | v1 |
| Layout Art Director (15) | `15_layout_art_director_v1.8.md` | v1.9 |
| Layout Designer (08) | `08_layout_designer_v3.14.md` | v3.19 |
| Cover Designer (13) | `13_cover_designer_v2.6.md` | v2.8 |

**Если в config указан `_v1.md` вместо актуального файла — СТОП, исправить конфиг.**

### 2. Temperature агентов

| Агент | Temperature |
|-------|-------------|
| Cleaner | 0.1 |
| Fact Extractor | 0.15 |
| Историк | 0.3 |
| **Ghostwriter** | **0.4** (было 0.6, снизили для точности) |
| Fact Checker | 0.1 |
| Literary Editor | 0.5 |
| Proofreader | 0.0 |

### 3. Max_tokens — не ограничивать

Все агенты должны иметь `max_tokens >= 32000` (Ghostwriter, Literary Editor, Proofreader, FactExtractor). Layout Designer — 64000. Не экономить на токенах — дешевле дать модели дописать, чем перегнать прогон.

### 4. Транскрипты на входе

- **TR1** (Phase A) — обязательный. Путь проверить.
- **TR2** (Phase B) — опциональный. Если есть, копировать в папку прогона.

### 5. Результат pre-flight

Сохранить `pre_flight_check_vXX.json` в папку прогона:

```json
{
  "timestamp": "...",
  "config_sha256": "...",
  "prompt_versions": {"ghostwriter": "v2.11", ...},
  "temperatures": {"ghostwriter": 0.4, ...},
  "transcripts": ["TR1_path", "TR2_path"],
  "status": "ready"
}
```

**Если хоть один пункт не соответствует — прогон НЕ запускать.**

## Stage 1 — Извлечение фактов

### Цель
Получить структурированную карту фактов (`fact_map.json`) из транскрипта интервью.

### Агенты и последовательность
1. **Cleaner (01)** — чистит ASR-транскрипт (пунктуация, разметка спикеров)
2. **Fact Extractor (02)** — извлекает факты, персон, события, места, цитаты, character_traits, gaps

### Вход
- `TR1` (основной транскрипт)

### Выход
- `cleaned_transcript.txt` — очищенный транскрипт
- `fact_map.json` — структурированные факты

### Что проверить
- Cleaner не обрезал текст (сравнить длину входа и выхода, должна быть ≈ +5%)
- Fact Extractor извлёк:
  - subject с полными данными (name, birth_year, death_year, birth_place)
  - ≥10 персон (родственники, знакомые)
  - ≥20 событий в timeline
  - ≥5 quotes (usable_in_book: true)
  - character_traits с категориями (в т.ч. metaphor для образных сравнений)
  - gaps (пробелы для интервьюера)

### Артефакты в collab
- `stage1_manifest.json` — версии промптов, хэши входов, времена
- `fact_map_vXX.json`

## Stage 2 — Написание книги (Phase A)

### Цель
Сгенерировать черновик книги из fact_map.

### Агенты и последовательность

1. **Историк (12)** — работает **последовательно ДО** Ghostwriter, генерирует исторические справки, которые подаются Ghostwriter на вход
2. **Ghostwriter (03)** — пишет текст всех глав + bio_data/timeline для ch_01
3. **Fact Checker (04)** — проверяет текст, цикл возврата

### Цикл возврата FC → Ghostwriter

Максимум **3 итерации**:
1. Ghostwriter пишет → FC проверяет
2. Если FC FAIL → Ghostwriter исправляет → FC проверяет
3. Если всё ещё FAIL → последняя попытка → FC
4. Если после 3-й итерации FAIL → **блокировать прогон**, диагностика

**Severity ошибок:**
- `critical` — блокирует прогон (фейковые факты, death_year при null, ch_01 без bio_data)
- `major` — требует исправления (галлюцинации-интерпретации, символизация слов)
- `minor` — предупреждение, не блокирует

### Вход
- `fact_map.json` (Stage 1)
- `historian_result.json` (параллельно)

### Выход
- `book_draft.json` — книга из 5 разделов: ch_01, ch_02, ch_03, ch_04, epilogue
  - ch_01 — `content: null`, `bio_data: {personal, education, military, awards, family}`, `timeline: [4-7 этапов]`
  - ch_02..ch_04 — `content` с абзацами и подсекциями ##
  - epilogue — 3-5 абзацев, без оценок

### Что проверить
- Все 5 глав есть в book_draft
- ch_01.bio_data содержит минимум 4 секции (personal, education/military, family, timeline)
- ch_01.timeline — 4-7 этапов с period/title/text
- ch_01.bio_data.family — ВСЕ прямые родственники из fact_map.persons
- ch_02 содержит подсекции ##
- gl.03 — плоская структура (без промежуточного "Ценности и установки")
- historical_notes[] — непустые, с title и content, chapter_id="ch_02"
- callouts[] — 6-7 штук, только в ch_03 и ch_04
- FC вердикт PASS после ≤3 итераций

### Артефакты в collab
- `stage2_manifest.json`
- `historian_result_vXX.json`
- `book_draft_vXX.json`
- `fc_report_iter1.json`, `fc_report_iter2.json`, `fc_report_iter3.json`

## Stage 3 — Шлифовка

### Цель
Доработать стиль и проверить орфографию/типографику.

### Агенты и последовательность

1. **Literary Editor (05)** — шлифует стиль, убирает AI-клише, усиливает голос рассказчика
2. **Proofreader (06)** — финальная вычитка (орфография, пунктуация, типографика)

### Proofreader — per-chapter режим

Если книга > 10,000 символов — вызывается **по одной главе**:
1. Первая глава генерирует `style_passport`
2. Последующие главы получают `style_passport` как вход + контекст стыков
3. Fallback только для упавшей главы, остальные вычитаны

### Вход
- `book_draft.json` (Stage 2)

### Выход
- `book_FINAL_stage3.json` — финальный текст Phase A

### Что проверить
- Все главы вычитаны (нет `is_modified: false` без причины)
- `bio_data` в ch_01 перенесён в финальный JSON (баг, который уже находили — bio_data терялся в Stage 3)
- style_passport сохранён
- Объём почти не изменился (±100 символов от Stage 2)

### Артефакты в collab
- `stage3_manifest.json`
- `book_FINAL_stage3_vXX.json`
- `liteditor_report_vXX.json`
- `proofreader_report_vXX.json` (style_passport внутри — отдельного файла нет)

## Phase B — Добавление нового материала

### Когда запускается
Если есть **второй транскрипт (TR2)** — например, дополнительное интервью с другим родственником.

### Цель
Вплести новый материал из TR2 в существующую книгу без переписывания всей структуры.

### Агенты и последовательность

1. **Incremental Fact Extractor** — запускает FE на TR2 с `existing_facts = fact_map` → мержит с дедупликацией (задача 007)
2. **Ghostwriter (03)** — `revision_scope.type: "content_addition"` — вплетает новый материал в главы ch_02, ch_03, ch_04
3. **Fact Checker (04)** — проверяет изменённые главы (`affected_chapters`)

### Цикл возврата FC → Ghostwriter Phase B

**Минимум 1 попытка, максимум 3 (обновлено в v32, было 2):**
1. Ghostwriter пишет дополнения → FC проверяет
2. Если FC FAIL — Ghostwriter получает конкретные ошибки → правит → FC проверяет
3. Если после 3-й итерации FAIL — **блокировать прогон**, диагностика

> **ВАЖНО:** До задачи 007 Phase B делала только 1 итерацию. После задачи 007 — нужна возможность перезапускать Ghostwriter при FAIL.

### Вход
- `book_FINAL_stage3.json`
- `TR2` (новый транскрипт)
- `fact_map.json` (будет обновлён Incremental FE)

### Выход
- `fact_map_phase_b.json` — обновлённый fact_map с новыми фактами из TR2
- `book_FINAL_phase_b.json` — книга с вплетённым новым материалом

### Что проверить
- Incremental FE добавил новые персоны/события из TR2 с `is_new: true`
- Ghostwriter расширил ch_02, ch_03, ch_04 — объём вырос на 20-50%
- epilogue и ch_01 НЕ трогались (scope_lock)
- Стыки старого и нового текста плавные (Literary Editor не запускается повторно — отвечает Ghostwriter)
- FC Phase B PASS

### Артефакты в collab
- `phase_b_manifest.json`
- `fact_map_phase_b_vXX.json` (с новыми элементами помеченными `is_new: true`)
- `book_FINAL_phase_b_vXX.json`
- `fc_phase_b_report.json`

## Stage 4 — Вёрстка (обзор)

### Цель
Превратить book_FINAL_phase_b в PDF книгу.

### Агенты и последовательность

1. **Photo Editor (07)** — анализирует фото, создаёт подписи
2. **Layout Art Director (15)** — планирует раскладку страниц
3. **Layout Designer (08)** — создаёт layout JSON (paragraph-ref архитектура)
4. **pdf_renderer** — рендерит PDF из layout + book_FINAL
5. **QA Layout (09)** — проверяет PDF (structural + visual)
6. **Cover Designer (13)** — создаёт обложку (отдельно)

### Ключевое правило

**Layout Designer больше не содержит текст абзацев** — только paragraph_ref (chapter_id + paragraph_id). pdf_renderer берёт текст из book_FINAL по id. Потеря текста физически невозможна.

### Детали
Stage 4 подробно описан в `pipeline-state.md`. Здесь только обзор.

## Ворота приёмки

### Gate 1 — Текст

**Что проверяет Даша:**
- Качество текста: факты, структура глав, отсутствие галлюцинаций
- Ключевые факты на месте (из чек-листа проекта)
- Подсекции ## во всех главах 02-04
- gl.03 плоская структура
- Эпилог сдержанный, без оценок

**Артефакты:**
- `*_FINAL_phase_b.txt` — читаемый текст
- `book_FINAL_phase_b.json` — структурированный
- `roles_checklist.md` — обязательно (см. ниже)

**Одобрение:** Даша читает текст, даёт OK или указывает конкретные проблемы.

### Gate 2a — Чистый текст в PDF

**Команда:** `--acceptance-gate 2a --text-only`

**Что проверяет:**
- Разбивка по страницам, заполненность (без дыр)
- Шрифты: название главы PT Sans Bold 17pt, подзаголовок PT Sans Bold 16pt, основной PT Serif 9.5pt
- Метки «Глава NN» перед названиями
- Формат A5 (148×210 мм)
- Layout Designer вызывается ОДИН РАЗ → layout JSON сохраняется для переиспользования

**Артефакты:**
- `*_gate2a_text_only.pdf`
- `*_gate2a_layout.json` (для переиспользования)
- `layout_text_approved.json` checkpoint

### Gate 2b — Текст + bio_data

**Команда:** `--acceptance-gate 2b --existing-layout LAYOUT --with-bio-block`

**Что проверяет:**
- ch_01 с полным bio_data: личные данные, образование, военная служба, награды, семья
- Timeline 4-7 этапов с вертикальной линией и точками-маркерами
- Секционные заголовки (ЛИЧНЫЕ ДАННЫЕ, ОБРАЗОВАНИЕ, ...) — золотые, 7pt, uppercase
- KeepTogether для заголовков секций (не отрываются от данных)

**Артефакты:**
- `*_gate2b_bio.pdf`
- `layout_bio_approved.json` checkpoint

### Gate 2c — + callout/справки/плейсхолдеры

**Команда:** `--acceptance-gate 2c --existing-layout LAYOUT --no-photos`

**Что проверяет:**
- Callout в ch_03 и ch_04 по образцу: PT Serif Italic 13pt, leading 20pt, линии сверху/снизу #c4a070, без фона, inset 10мм от полей
- Подпись «— из воспоминаний семьи» — PT Serif Italic 8pt справа, #8b7355
- Исторические справки в ch_02: фон #f0ebe0, левая черта 2pt #c4a070, лейбл «ИСТОРИЧЕСКАЯ СПРАВКА» 6.5pt, текст PT Serif Italic 9pt #5a4a38
- Плейсхолдеры фото на chapter_start (ch_02, ch_03, ch_04) — серые рамки с подписью «[ФОТО — начало главы]»
- Плейсхолдеры фотораздела в конце — по правилам ориентации (vertical 1/страница, horizontal 2/страница стопкой)

**Артефакты:**
- `*_gate2c_full.pdf`
- `layout_full_approved.json` checkpoint

### Gate 3 — Реальные фото

**Команда:** `--acceptance-gate 3 --existing-layout LAYOUT --photos-dir PHOTOS`

**Что проверяет:**
- Photo Editor создал подписи
- Art Director выбрал фото для chapter_start (приоритет portrait)
- Вертикальные фото — 1 на страницу full-width
- Горизонтальные — pair_stack (2 на страницу стопкой)
- Микс запрещён (vertical + horizontal на одной странице — нельзя)

**Артефакты:**
- `*_gate3_photos.pdf`
- `photos_approved.json` checkpoint

### Gate 4 — Обложка

**Команда:** `--acceptance-gate 4 --existing-layout LAYOUT --photos-dir PHOTOS --with-cover`

**Что проверяет:**
- Cover Designer выбрал portrait фото (приоритет portrait > group > event)
- AI-портрет сгенерирован через Replicate (img2img с реальным фото как reference)
- 3-зонная композиция (top/middle/bottom)
- death_year=null → годы «род. ГГГГ», иначе «ГГГГ — ГГГГ»
- Форзаца нет, после обложки сразу TOC

**Артефакты:**
- `*_gate4_final.pdf` — финальная книга
- `cover_portrait.webp`
- `cover_approved.json` checkpoint

**Шаблон:**

```markdown
# Roles checklist: run vXX

## Pre-flight
- Config_sha256: ...
- Дата: ...
- Транскрипты: TR1 (XXX симв.), TR2 (если есть)

## Phase A (Stage 1-3)

| # | Роль | Этап | Отработала? | Промпт (версия) | Temperature | Время | Вердикт |
|---|------|------|------|------|------|------|------|
| 01 | Cleaner | S1 |   | 01_cleaner_v1 | 0.1 |   |   |
| 02 | Fact Extractor | S1 |   | 02_fact_extractor_v3.3 | 0.15 |   |   |
| 12 | Историк | S2 |   | 12_historian_v3 | 0.3 |   |   |
| 03 | Ghostwriter | S2 |   | 03_ghostwriter (v2.11) | 0.4 |   | итераций: N |
| 04 | Fact Checker | S2 |   | 04_fact_checker (v2.4) | 0.1 |   | итераций: N, финал: PASS/FAIL |
| 05 | Literary Editor | S3 |   | 05_literary_editor_v3 | 0.5 |   |   |
| 06 | Proofreader | S3 |   | 06_proofreader_v1 | 0.0 |   | per-chapter: Y/N |

## Phase B (если есть TR2)

| # | Роль | Этап | Отработала? | Промпт | Temperature | Время | Вердикт |
|---|------|------|------|------|------|------|------|
| 02 | Fact Extractor (inc) | PB |   | 02_fact_extractor_v3.3 | 0.15 |   |   |
| 03 | Ghostwriter | PB |   | 03_ghostwriter (v2.11) | 0.4 |   | итераций |
| 04 | Fact Checker | PB |   | 04_fact_checker (v2.4) | 0.1 |   | итераций, финал |

## Flags (что требует внимания)

- [ ] Все роли отработали (Отработала = Y для всех строк)
- [ ] Версии промптов соответствуют pre-flight check
- [ ] Temperature соответствуют
- [ ] FC финальный вердикт PASS на Stage 2 и Phase B
- [ ] Артефакты в collab/runs/

## Замечания для ревью Даши
(если есть особенности, отмечаем здесь)
```

### Правило

**Без заполненного roles_checklist — Даша НЕ смотрит PDF.**

Разработчик присылает PDF вместе с checklist. Если checklist неполный или показывает пропущенного агента — останавливаем ворота, чиним, перезапускаем.

## Процедура при падении агента

### 1. Если упал Cleaner / Fact Extractor (Stage 1)
- Остановить прогон
- Проверить: размер входного файла, API ключ, max_tokens
- Исправить и перезапустить Stage 1 заново

### 2. Если упал Историк (Stage 2)
- **Не пропускать молча.** Если отработал — записать в manifest. Если упал — блокировать или явно документировать что его нет
- Если `--skip-historian` намеренно — это должно быть в manifest, чтобы не забыть при анализе PDF

### 3. Если FC FAIL после 3 итераций (Stage 2)
- Blокировать прогон
- Читать последний fc_report, смотреть что именно Writer не может исправить
- Варианты: усилить промпт Writer, снизить temperature, исправить fact_map вручную

### 4. Если Proofreader упал (Stage 3)
- Fallback на текст Literary Editor (без proofread)
- Но отметить в checklist что Proofreader отвалился

### 5. Если Phase B FC FAIL после 2 итераций
- Blокировать прогон
- Смотреть что Writer не может исправить в новом материале

### 6. Общее правило

**Лучше STOP и починить, чем продолжить с пропущенным агентом.**

Если проскочили агента — не делать вид что всё ок. Фиксировать в checklist и решать: можно ли идти дальше или нужно перезапустить с этого этапа.
