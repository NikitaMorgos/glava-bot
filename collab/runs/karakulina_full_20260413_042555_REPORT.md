**Проблема:** В каждом прогоне Корректор возвращает пустой ответ (`JSON parse error: line 1 col 1`). Это не случайность — воспроизводится стабильно.

**Предположение:** Промпт требует вернуть полный текст книги (~32K токенов), но claude-sonnet при `temperature=0.0` иногда «зависает» на пустом ответе при больших выходах. Либо промпт содержит инструкцию, которую модель не может выполнить при данном объёме.

**Рекомендации:**
- Проверить промпт `06_proofreader_v1.md` — возможно, он ожидает другой формат входа
- Попробовать разбить Корректора на глав-по-одной (итерируемый вызов)
- Или снизить ответственность Корректора: пусть возвращает только список правок (diff), а не полный текст

### 🟡 Важно: Phase B FC FAIL — нарастающие ошибки при итерациях

**Проблема:** После Phase B Ghostwriter FC возвращает FAIL, и при правке итерация 2 ухудшается (8 → 10 ошибок). Писатель при правке вводит новые проблемы, пытаясь исправить старые.

**Анализ ошибок итерации 2:**
- 8× `distortion` в ch_02 — в основном неточные количественные формулировки и маркеры уверенности
- 1× `hallucination` в ch_03 — интерпретации без источника
- 1× `hallucination` в ch_04 — аналогично

**Рекомендации:**
1. **Ограничить итерации Phase B FC до 1** — вторая итерация в текущей архитектуре только портит. Лучше FAIL с 8 ошибками, чем FAIL с 10.
2. **Настроить FC v2.1 для Phase B** — сейчас FC проверяет весь текст книги, включая главы от Stage2 (которые уже прошли FC PASS). Нужен режим «проверять только изменённые главы».
3. **Промпт Писателя для правок** — при `call_type=revision` в Phase B Писатель получает список ошибок и правит главу целиком, что приводит к регрессии. Нужна точечная правка по `fix_instruction`.

### 🟡 Важно: Phase B без Фактолога

**Проблема:** Фактолог v3.3 не запускается на TR2 перед Phase B. Новые факты из интервью Татьяны (даты, топонимы, персоны) поступают в Писателя без нормализации. Это приводит к distortion-ошибкам типа «несколько сестёр» вместо «две».

**Рекомендация:** Добавить шаг «Incremental FactExtractor» перед Phase B:
- Запускать ФЭ только на новом транскрипте
- Мержить новые факты в существующий fact_map (по `id`, без перезаписи)
- Передавать обогащённый fact_map в Phase B Ghostwriter

Это устранит большинство distortion-ошибок Phase B без потери скорости.
