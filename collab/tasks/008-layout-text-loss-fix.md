# 008 — Потеря текста в Layout Designer: переход на ссылочную архитектуру

Статус: new

Автор: Даша

## Контекст

В прогоне `karakulina_full_20260414_062711` обнаружена потеря 64% текста. Книга содержит 14,145 символов, но в PDF попало только 4,327. Причина: Layout Designer (роль 08) при генерации layout JSON **пересказывает** текст абзацев вместо дословного копирования. На 11 текстовых страницах в среднем по 400 символов вместо ~1,300.

Доказательство:
- `karakulina_book_FINAL_phase_b_20260414_121019.json` → 14,145 символов
- `karakulina_stage4_layout_iter2_20260414_175419.json` → 4,327 символов в paragraph.content
- `karakulina_FINAL_with_photos_20260414.pdf` → визуально подтверждено: ch_02 обрывается на работе в поликлинике, нет про Татьяну, смерть мужа, последние годы

Артефакты: `collab/runs/karakulina_full_20260414_062711/`

## Проблема

Layout Designer получает полную книгу (14K символов) и page_plan от Арт-директора. При генерации layout JSON он должен разместить текст по страницам, но вместо копирования исходных абзацев **генерирует собственные сокращённые версии**. Это LLM-поведение — при temperature 0.25 модель парафразит и сжимает.

Текущая архитектура:
```
book_FINAL.json (14K) → Layout Designer → layout.json (paragraph.content = пересказ 4K) → pdf_renderer → PDF (4K)
```

pdf_renderer честно рендерит то, что дал Layout Designer. Проблема не в рендерере.

## Решение: ссылочная архитектура

Изменить архитектуру так, чтобы pdf_renderer подтягивал текст напрямую из книги, а Layout Designer отвечал только за структуру страниц.

### Новая архитектура:
```
book_FINAL.json (14K) → Layout Designer → layout.json (paragraph_ids, позиции, типы) → pdf_renderer (подтягивает текст из book_FINAL по id) → PDF (14K)
```

### Что меняется в Layout Designer (промпт 08):

**Было:**
```json
{
  "page_number": 7,
  "content_type": "text_only",
  "elements": [
    {
      "type": "paragraph",
      "content": "Пересказанный текст абзаца..."
    }
  ]
}
```

**Станет:**
```json
{
  "page_number": 7,
  "content_type": "text_only",
  "chapter_id": "ch_02",
  "elements": [
    {
      "type": "paragraph",
      "paragraph_id": "p_02_005"
    },
    {
      "type": "paragraph",
      "paragraph_id": "p_02_006"
    }
  ]
}
```

Layout Designer больше не пишет текст — только указывает, какие абзацы на какой странице. Фото, callout-ы, исторические справки — тоже по id.

### Что меняется в pdf_renderer.py:

- Получает на вход layout.json + book_FINAL.json
- Для каждого `paragraph_id` в layout берёт текст из `book_FINAL.chapters[].paragraphs[]`
- Для callout-ов — из `book_FINAL.callouts[]` по id
- Для исторических справок — из `book_FINAL.historical_notes[]` по id
- Если paragraph_id не найден в книге → ошибка (не молчаливый пропуск)

### Что меняется в QA (промпт 09):

- Добавить проверку: все paragraph_id из layout ссылаются на реальные абзацы в книге
- Добавить проверку: все абзацы книги присутствуют в layout (нет потерянных)

## Преимущества

1. **Невозможно потерять текст** — LLM не отвечает за передачу текста, только за позиционирование
2. **Меньше токенов** — Layout Designer не возвращает 14K текста, только структуру (~2-3K)
3. **Быстрее** — меньше генерируемых токенов = быстрее ответ
4. **Дешевле** — меньше output tokens = меньше стоимость
5. **Детерминированно** — текст из книги берётся кодом, а не LLM

## Что не меняется

- Промпт Арт-директора (15) — page_plan остаётся как есть
- Промпт Писателя (03) — формат книги не меняется
- Stage1–3, Phase B — без изменений
- Обложка — без изменений

## Чеклист проверки

- PDF содержит 100% текста из book_FINAL (символ-в-символ)
- Все абзацы книги присутствуют в PDF
- Layout Designer возвращает paragraph_id, а не content
- pdf_renderer подтягивает текст из book_FINAL по id
- Если paragraph_id не найден → ошибка, а не молчаливый пропуск
- QA проверяет полноту: все абзацы книги → все страницы PDF
- Тест на Каракулиной: PDF содержит 14,145 символов
