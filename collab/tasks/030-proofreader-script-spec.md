## Контекст

Из holistic-аудита: Proofreader (T=0.0) — самый низко висящий фрукт для скриптования. Промпт делает 6 проверок:

| # | Проверка | Скриптуемо? |
|---|---|---|
| 1 | Орфография | ✅ Hunspell / Languagetool / pyspellchecker |
| 2 | Пунктуация | ✅ Languagetool (rules-based) |
| 3 | Типографика (« », —, ё, неразрывные пробелы) | ✅ regex/Unicode полностью |
| 4 | Единообразие (имена, даты, аббревиатуры) | ✅ scan + canonicalize |
| 5 | Грамматика | 🟡 Languagetool частично |
| 6 | Markdown | ✅ markdown-it / mistune парсер |

5/6 — почти полностью скриптом. Только грамматика частично требует LLM (если нужна высокая точность).

Output Proofreader (что должен генерировать скрипт):
- `corrections[]` — список исправлений с типом (spelling/punctuation/typography/consistency/grammar/markup), before/after, rule
- `style_passport` — реестр имён, дат, кавычек, аббревиатур
- `summary` — статистика по типам

## Три design-вопроса

### Q1: Выбор основного инструмента

| Опция | Coverage | Зависимости | Прогон |
|---|---|---|---|
| **A. Languagetool (jvm)** | 95% от LLM | Java JDK 8+ (~300MB), CPU+RAM на сервере | ~3-5 сек на главу |
| **B. pyspellchecker + кастом regex** | 60% от LLM | Чистый Python, нет внешних зависимостей | <1 сек |
| **C. pylanguagetool через public API** | 95% от LLM | Зависимость от внешнего сервиса, rate limit | 2-5 сек на главу + сетевая латентность |
| **D. Гибрид: regex+pyspellchecker для типографики/орфографии + Languagetool по запросу для грамматики** | 80% | Java только если нужна грамматика | 1-3 сек |

**Что нужно знать:**
- Есть ли на сервере Java JDK?
- Допустимы ли внешние API-вызовы для proofreading (приватность данных клиента)?
- Какое время прогона приемлемо? (сейчас Proofreader-агент = 30-60 сек на главу)

**Default Опуса (если решат не вы):** D — гибрид. Большинство ошибок (типографика, орфография, единообразие) ловятся без Java. Грамматика — опциональный enrichment, может быть отложен.

### Q2: Acceptance criteria для замены LLM

Скриптный proofreader не будет ловить 100% того что ловит LLM. Какой % считается достаточным для production-замены?

| Опция | Что значит |
|---|---|
| **A. ≥ 95% совпадения** | Скрипт почти равен LLM. Требует Languagetool. |
| **B. ≥ 80% + явные warnings** | Скрипт ловит big-impact (орфография, типографика), редкие случаи флагит как warning «возможно требует LLM-проверки». |
| **C. ≥ 70% + LLM-fallback** | Скрипт ловит то что может, остальное передаёт LLM-агенту. Гибрид run-time. |

**Что нужно знать:**
- Какой % real-world Proofreader-corrections — типографика+орфография (точно скриптуется), а какой — нюансы грамматики (LLM лучше)?
- Готова ли семья получить книгу с одной-двумя пропущенными опечатками если за это пайплайн в 10 раз дешевле и быстрее?

**Default Опуса:** B — 80% + warnings. Для книг-биографий орфография и типографика важнее редких грамматических нюансов; редкие случаи можно поднимать как warning человеку-редактору.

### Q3: Style passport — кто отвечает за консистентность имён?

Сейчас Proofreader **создаёт** style_passport на Phase A: реестр имён (Иванов Пётр Сергеевич / Пётр Сергеевич / Пётр / папа), формат дат, кавычки. Phase B сверяется с уже созданным.

При скриптовании:
| Опция | Где создаётся style_passport |
|---|---|
| **A. Из fact_map** | Имена + relations берутся из `fact_map.persons[]`, форматы дат — из ASR/конвенций. Полностью детерминируемо. |
| **B. NER-эвристика** | spaCy / natasha извлекает named entities из текста, скрипт canonicalize'ит. Требует NER-зависимости. |
| **C. LLM-агент по запросу** | Скрипт делает обвязку, для генерации style_passport вызывает LLM. Гибрид. |

**Default Опуса:** A — из fact_map. У нас уже есть структурированный источник истины для имён. Кавычки и тире — Unicode-rules (pyrtypograph, smartypants для русского). Даты — детектор форматов на regex.
