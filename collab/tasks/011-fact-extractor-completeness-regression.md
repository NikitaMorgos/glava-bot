## Контекст

В прогоне v35 Fact Extractor v3.4 на объединённом входе TR1 (20KB) + TR2 (49KB) = 69KB **не извлёк** критически важные факты из TR2, хотя в предыдущих прогонах (v28, v32) эти факты попадали в fact_map.

**Факты, которые потерялись при извлечении:**
- Пианино / шуба (эпизод: Валентина продала шубу из Венгрии, купили пианино дочери) — 0 упоминаний
- Огурцы / чемодан / Молдавия (эпизод с зятем Маргосем) — 0
- «Посидеть на дорожку» (семейная традиция) — 0
- Немецкий сервиз / фарфор (эпизод с покупкой сервиза за месячную зарплату) — 0
- Сахалин (развилка судьбы мужа) — 0 (!)

**Ещё серьёзнее — неполная структура persons:**
- В `fact_map_full_v35.json` 11 персон, но **дочь Татьяна отсутствует**
- Есть сын Валерий, есть зятья (Маргось, Кужба — мужья Татьяны), но **самой Татьяны нет**
- Невозможная структура: зять есть, дочери нет

**Отдельная аномалия:** в книге ch_02 написано «Дмитрию предложили либо Сахалин, либо увольнение», но в fact_map события с Сахалином нет. Значит GW либо взял факт напрямую из транскрипта (нарушение архитектуры), либо из фоновых знаний (галлюцинация).

**В v28 и v32 эти факты были извлечены.** Значит произошла регрессия либо:
- В промпте FE v3.4 (при добавлении reasoning/confidence для топонимов)
- В коде подачи объединённого входа (TR1 + TR2 склейка)
- В обработке длинного контекста (69KB — длинно, но не запредельно)

### Диагностика перед реализацией

#### Что реально подавалось FE v3.4 на вход в v35

Из `manifest_s1_v35.json`:
- **Входные файлы:** TR1 = `karakulina_valentina_interview_assemblyai.txt`, TR2 = `karakulina_meeting_transcript_20260403.txt`
- **Схема сборки:** оба транскрипта **конкатенируются** в `combined` → один вызов Cleaner → один вызов FE
- **Разделитель:** `=== ИСТОЧНИК 1: <имя файла> ===` / `=== ИСТОЧНИК 2: <имя файла> ===`
- **FE вызывается один раз** на очищенном объединённом тексте; разбивки на два прохода нет
- **Единственный `interview.id`:** `"int_001"` — оба транскрипта атрибутируются одному источнику

Размеры из `pre_flight_check_v32.json` (те же файлы):
- TR1: 20 675 байт, TR2: 48 601 байт → ~69 KB сырой вход в Cleaner

#### Сравнение v32 vs v35

| Метрика | v32 (FE **v3.3**) | v35 (FE **v3.4**) |
|---------|-------------------|-------------------|
| Persons | ~20 (из спека задачи) | **13** (факт: `fact_map_full_v35.json`) |
| Timeline events | ~32+ | **28** |
| Татьяна в persons | ✅ есть | ❌ **отсутствует** |
| Племянники (TR2) | ✅ 4 записи | ❌ **0** |
| Пианино/шуба | ✅ | ❌ |
| Сахалин | ✅ | ❌ |

Ключевое наблюдение: в v35 есть **событие «Рождение дочери Татьяны»** в timeline, но **нет персоны Татьяна** в persons[]. Это признак усечения вывода: FE сгенерировал часть JSON и остановился до заполнения полного persons[].

#### Корневая причина: токенный бюджет выхода FE

v3.4 добавил к каждой персоне поля `asr_variants`, `reasoning`, `confidence` (~100–200 доп. токенов на запись), а к каждой локации — обязательное поле `reasoning` (1–2 предложения). На 69 KB входа FE v3.3 укладывался в 32K выходных токенов. FE v3.4 с теми же данными превышает лимит: вывод обрезается в середине `persons[]`, хвост транскрипта (TR2) и завершение массива теряются.

**Вывод: регрессия вызвана увеличением verbosity промпта v3.4 при фиксированном `max_tokens: 32000`, а не изменением логики подачи входа.**

Дополнительная уязвимость: Cleaner также получает ~69 KB с `max_tokens: 32000`. Если Cleaner компактифицирует текст недостаточно (особенно TR2 в формате meeting transcript с диаризацией), FE получает ещё более длинный «cleaned» текст.

### [TECH] Технические флаги

**[TECH-1] Корневая причина верифицирована на сервере: семантическая регрессия FE v3.4**

Данные с сервера (run.log + grep по cleaned transcript):
- FE output_tokens = **17 787** из лимита 32 000 → усечения нет
- Cleaner: 39 301 → 41 222 chars → TR2 не обрезан
- cleaned transcript содержит: пианино ×1, Сахалин ×1, сервиз ×1, Татьяна ×80, дорожку ×2

Все пропавшие факты были в FE-входе. FE v3.4 получил полный текст и выдал полный JSON — но не извлёк эти факты. Это поведенческая/семантическая регрессия, не truncation.

Два конкретных механизма:

**Механизм A — over-application Rule 8.1 (narrator ≠ persons):**
Рассказчик = "Татьяна Каракулина (дочь)". Rule 8.1 запрещает добавлять narrator в persons. FE v3.4 при построении `reasoning` для canonical name (новое поле) увидел совпадение имени рассказчика с дочерью и не создал person-запись. FE v3.3 в том же прогоне (апрель 13) корректно различал рассказчика-как-narrator и рассказчика-как-персонаж.

**Механизм Б — reasoning overhead on dense text:**
TR1 в cleaned output — одна плотная строка-конспект с ~30 фактами подряд. Фраза "Дмитрию предложили или на Сахалин, или уходить из армии" встроена внутрь этого потока. FE v3.4 с обязательным `reasoning` per item тратит "внимание" на главные биографические факты и пропускает sub-детали в плотных блоках. Аналогично — пианино/сервиз из длинного TR2-монолога без явных дат.

**[TECH-2] Отсутствие лога `truncated` для FE — нужно добавить профилактически**
В `run_fact_extractor` нет проверки `output_tokens >= max_tokens - 10` (в `run_cleaner` она есть). Добавить независимо от фикса — для диагностики будущих регрессий.

**[TECH-3] Rule 8.1 требует уточнения для случая narrator-is-also-subject-family**
Текущая формулировка Rule 8.1 не покрывает случай, когда рассказчик — прямой родственник и ключевой персонаж биографии. Нужно явно добавить: "Если рассказчик упомянут в контексте биографии как персонаж (дочь, сын, муж) — создаёт ОБЯЗАТЕЛЬНУЮ запись в persons с relation_to_subject".

**[TECH-4] Единый `interview.id` для объединённого входа нарушает атрибуцию**
Все факты из TR1 и TR2 помечаются `source_interview_id: "int_001"`. FC и GW не могут различить источник. При Варианте B (два прохода) это устраняется автоматически.

**[TECH-5] Вариант A достаточен для фикса Татьяны и Сахалина, Вариант B нужен для атрибуции**
Оба механизма регрессии (Rule 8.1 + reasoning-плотный текст) устранимы промптово (Вариант A). Вариант B (два прохода) нужен отдельно для корректного `source_interview_id`, но не является обязательным условием для v36.

**[TECH-6] v3.4 отсутствует в `prompts/CHANGELOG.md`**
CHANGELOG содержит записи до v3.3 (2026-04-12). Запись для v3.4 (asr_variants/reasoning/confidence для persons и locations) отсутствует. Нужно восстановить ретроспективно.

**[TECH-7] Регрессионный тест: нужен smoke-уровень без LLM**
Спек предлагает тест запускающий Stage 1 с LLM (~$0.5–1). Для `pytest` это неприемлемо. Разделить: (а) smoke — проверить список ключевых слов в существующем `fact_map_full_v35.json` (бесплатно, секунды); (б) full-integration — запуск Stage 1 только перед деплоем.

**[PRODUCT-2] Критерий регрессионного теста — отказ от числовых порогов.**

Числовые критерии («≥20 persons», «≥18 событий») **не универсальны**: они верны только для проекта Каракулиной. Для героя-одиночки или молодого героя 20 персон может быть нереально много, и тест ложно отклонит нормальный fact_map.

**Правильный критерий: соответствие транскрипту.**

Не «должно быть ≥N», а «всё что упомянуто в транскрипте → должно попасть в fact_map»:
- Если в транскрипте есть имя «Татьяна» → она обязана быть в `persons[]`
- Если упомянут «Сахалин» → событие обязано быть в `timeline[]` или фигурировать в описании
- Если упомянут «пианино» → должно быть в фактах/цитатах/character_traits

**Реализация:**

```yaml
# tests/regression_checks/karakulina_fact_completeness.yaml
required_terms_in_transcript:
  - term: "Татьяна"
    must_appear_as: "person.name OR person.aliases"
  - term: "Сахалин"
    must_appear_as: "event.title OR event.description"
  - term: "пианино"
    must_appear_as: "event.description OR trait.detail"
  - term: "Нинвана"
    must_appear_as: "person.name"
```

**Алгоритм:**
1. Прочитать транскрипт
2. Найти упоминания каждого `term`
3. Если найден в транскрипте → проверить что есть в fact_map по указанному пути
4. Если в транскрипте нет — пропустить, не флагать

**Работает для любого проекта:** для каждого героя свой `required_terms.yaml` с ключевыми словами из его транскриптов. Числовых порогов нет.

**Решение Даши:** ✅ согласна с подходом «упомянутое → извлечено», без чисел.

**[PRODUCT-3] Регрессионный тест — hard gate без override?**

Моё мнение: **hard gate с возможностью override через флаг `--allow-regression-fail` + обоснование в run_manifest**. Симметрично нашему решению по FC FAIL. Это даёт защиту по умолчанию, но не блокирует когда ты осознанно принимаешь риск.
