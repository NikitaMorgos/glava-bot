## Контекст

В task 014 был создан детерминированный Name Normalizer (`scripts/normalize_named_entities.py`) — он сливает дубликаты persons/locations в fact_map по совпадению имён + positional windows ±300 chars в транскрипте + порогу overlap ≥ 30%.

На пилоте Каракулиной v36 (2026-04-28) Name Normalizer показал критичную регрессию.

### Доказательство регрессии

В `karakulina_fact_map_full_v36b.json` запись `person_004 "Полина Амельченко"` имеет:

```json
{
  "id": "person_004",
  "name": "Полина Амельченко",
  "relation_to_subject": "старшая сестра",
  "aliases": ["тётя Поля", "Рудая Пелагея Алексеевна", "Марфа", "Дуня"]
}
```

Из этих aliases:
- ✅ `"тётя Поля"` — корректный alias старшей сестры Полины
- ❌ `"Рудая Пелагея Алексеевна"` — это **мать** Валентины, отдельная persona в fact_map (`person_002`), `relation_to_subject: "мать"`
- ❌ `"Марфа"`, `"Дуня"` — это **прабабушка** (мать матери Валентины), упомянутая в транскрипте без точного имени («может Марфа была или Дуня»)

То есть **три разных женщины с разными `relation_to_subject` склеены в одну запись.**

Все три downstream-агента (GW, Historian, FC) читают эту запись и считают, что у Полины-сестры есть второе имя «Рудая Пелагея Алексеевна», что она же — Марфа, и т.д. Это загрязняет всю цепочку.

### Почему текущий алгоритм это допустил

В транскрипте Татьяна обсуждает родственников по линии бабушки (мать Валентины) и тётю Полю в близких контекстах — фамилия «Рудая» обсуждается рядом с тётей Полей (искали девичью фамилию Полины). Имена «Марфа» и «Дуня» обсуждаются как варианты имени матери матери, но текстуально близко к разговору про сестёр.

Текущий Name Normalizer считает overlap позиций упоминаний и сливает по близости. Семантика родства в алгоритме **не учитывается**.

## Спек

### Системное правило

Persons из fact_map сливать в одну запись **только если ВСЕ условия выполнены**:

1. **Совпадение `relation_to_subject`** (с учётом синонимов): «сестра» = «старшая сестра» = «младшая сестра» = «двоюродная сестра» — сливаемы между собой как «сестра». Но «сестра» ≠ «мать» ≠ «бабушка» ≠ «тётя» — никогда не сливаемы, независимо от позиций.
2. **Совпадение или вариация канонического имени** (текущая логика positional + overlap остаётся, но как один из факторов).
3. **Отсутствие явного контекста разделения** — если в транскрипте есть фраза «у меня были две тёти, одна Маня, другая Поля» — Маня и Поля никогда не сливаются, даже если близко по позициям.

### Что нужно изменить в `scripts/normalize_named_entities.py`

1. **Добавить guard перед слиянием:** если `relation_to_subject` двух persons различаются по семантике (после нормализации модификаторов вроде «старшая/младшая/двоюродная»), слияние **запрещено**, независимо от positional overlap и совпадения имён.
2. **Семантические группы родства:**
   - parent: отец / мать
   - sibling: брат / сестра (любой модификатор)
   - spouse: муж / жена
   - child: сын / дочь
   - in-law-spouse: зять / невестка / свекровь / свёкор / тёща / тесть
   - grandparent: дедушка / бабушка / прадед / прабабушка
   - grandchild: внук / внучка / правнук / правнучка
   - extended: тётя / дядя / племянник / племянница / двоюродный
   - other-family: золовка / деверь / шурин / своячениц-
   - non-family: коллега / врач / сосед / друг / etc.
3. **Слияние разрешено только внутри одной семантической группы.** Между группами — никогда.
4. **Не объединять по `aliases` если у aliases разный relation:** при добавлении alias из другой persons-записи — если её relation в другой группе, alias не добавляется, сигнализируется как отдельная сущность.

### Какой результат ожидается

На том же входе (v36 транскрипт + текущая v36 fact_map) после прогона нового Name Normalizer:

- `person_004 "Полина Амельченко"`: aliases = только корректные («тётя Поля»; возможно «Поля», «Поличка», варианты с фамилией). Без «Рудая Пелагея Алексеевна» и без «Марфа», «Дуня».
- `person_002 "Рудая Пелагея Алексеевна"`: остаётся отдельной записью с relation = «мать», без склейки с сестрой.
- Если в transcript есть упоминание «Марфа/Дуня» как варианты имени **прабабушки** — должна возникнуть отдельная запись (или `log_only_gap` от Completeness Auditor) с relation = «прабабушка», а не приписка к Полине.

### Диагностика перед реализацией

**Корневая причина подтверждена.** Точка отказа — функция `_normalize_entities()` в `scripts/normalize_named_entities.py`, цикл строк 447–504.

Шаги слияния (упрощённо):
1. Для каждой persons-записи генерируется `match_a` — набор вариантов имени (включая aliases и диминутивы через словарь `DIMINUTIVES`).
2. `DIMINUTIVES["пелагея"] = {"пелагея", "поля", "полина"}` → для `person_004 "Полина Амельченко"` (сестра, alias «тётя Поля») `match_a` включает `{"полина", "поля", "пелагея"}`.
3. Для `person_002 "Рудая Пелагея Алексеевна"` (мать) `match_b` включает `{"рудая", "пелагея", "алексеевна", "поля", "полина"}` (через тот же `DIMINUTIVES["пелагея"]`).
4. `common_variants = {"поля", "полина", "пелагея"}` — три значимых токена, все ≥ `MIN_NAME_LEN=3`, ни один не в `STOP_TOKENS`. Условие `len(significant_common) >= 2` выполнено → сливаем.
5. `_positions_overlap_ratio()` находит позиционное пересечение (транскрипт: тётя Поля обсуждается рядом с девичьей фамилией Рудая, так как ищут фамилию сестры) → `overlap ≥ 0.30` → слияние происходит.
6. `_merge_person_records()` добавляет `"Рудая Пелагея Алексеевна"` в aliases person_004.

**Имена «Марфа/Дуня»** попадают в aliases Полины той же цепочкой: в транскрипте обсуждается «имя прабабушки — может Марфа или Дуня» в разговоре про родственников матери, позиционно близко к упоминаниям тёти Поли. Нет отдельной persons-записи для прабабушки → нет relation_to_subject = «прабабушка» → guard по текущей логике вообще не срабатывает, имена просто приписываются ближайшей Полине.

**Что не проверяется в текущем коде:** `relation_to_subject` полностью игнорируется в `_normalize_entities()`. Поле присутствует в persons-записях, но ни в одном из условий слияния (строки 447–504) оно не читается.

### [TECH] Технические флаги

**[TECH-1] Коллизия через DIMINUTIVES — системный риск, не единичный случай.**
`DIMINUTIVES["пелагея"]` = `{поля, полина, пелагея}`. Любые две persons-записи с именами из этого кластера могут слиться, если упоминаются в тексте рядом. Для семьи, где несколько женщин, — частый паттерн. Нужен guard, а не расширение DIMINUTIVES.

**[TECH-2] Нормализация `relation_to_subject` — нетривиальная задача.**
Поле содержит произвольные строки: `"старшая сестра"`, `"дочь тёти Шуры"`, `"золовка (сестра мужа)"`. Нужна нормализация перед сравнением групп:
- Базовый подход: keyword-in-string matching по каждой группе (не exact match, не full NLP).
- Если `relation_to_subject` пустой для одной из записей — консервативно считать её группу `"unknown"`, не сливать с известными группами.
- Compound relations (`"дочь тёти Шуры"`) — нужно матчить на несколько групп. Правило: если первое ключевое слово (дочь) → group `child`, но второе (тёти) → `extended`. Это уже не одна group → блокировать слияние с pure `child` или pure `sibling`.

**[TECH-3] «Марфа/Дуня» останутся незакреплёнными после фикса.**
Guard не создаёт отдельные persons-записи для упомянутых без id имён. Если прабабушки нет в fact_map как отдельная запись — Марфа/Дуня не будут приписаны никому. Это корректное поведение (не ложный merge), но Completeness Auditor может создать отдельную запись с `relation="прабабушка"`. Задача 016 это не затрагивает напрямую.

**[TECH-4] Rejected-merge log — добавить обязательно.**
Спек требует логировать каждое отказанное слияние. В текущем коде `merged_pairs_log` содержит только успешные слияния. Нужно добавить `rejected_pairs_log` с полем `"blocked_reason": "relation_group_mismatch"`. Это важно для отладки на новых клиентах, где relation_to_subject может быть пустым.

**[TECH-5] Signature `normalize_named_entities(fact_map, transcript)` — изменений нет.**
Guard добавляется внутри `_normalize_entities()`, не меняет публичный интерфейс.

**[TECH-6] `DIMINUTIVES` не исправляем.**
Словарь корректен — `пелагея/поля/полина` действительно диминутивы. Проблема не в словаре, а в отсутствии semantic guard. Не трогать.

### [PRODUCT] Продуктовые флаги

**[PRODUCT-1] Что делать если `relation_to_subject` пустой у одной из записей?**
Две options: (a) консервативно блокировать слияние (не знаем кто это — лучше две записи, чем одна неправильная); (b) разрешать только если empty-record имеет уникальное имя без коллизий.
Рекомендация: **блокировать с предупреждением**, если known-group запись слипается с empty-relation записью. Оба случая логировать.

**[PRODUCT-2] Что делать если `relation_to_subject` содержит конфликтующие keyword (как «дочь тёти Шуры»)?**
Такая строка содержит `"дочь"` (group: child) и `"тёти"` (group: extended). Текущий спек говорит «сестра» ≠ «мать» — но что если relation неоднозначен? Рекомендация: если строка матчит ≥ 2 разных групп — считать её `"ambiguous"`, не сливать ни с чем. Логировать как отдельный rejected_reason.

**Оценка сложности:** `s`
**Оценка риска:** `medium` — guard может заблокировать легитимные слияния при непоследовательном FE-формировании `relation_to_subject` (у разных FE-прогонов поле может быть «сестра» vs «старшая сестра» vs пусто). Митигация: регрессионный тест на v36 данных + проверка что existing correct merges (Каракулина = Валентина = Валя) не ломаются.
