# 009 — Протокол ворот приёмки: послойная проверка PDF

Статус: dasha-review
Приоритет: высокий
Автор: Даша
Дата: 2026-04-14

## Контекст

Текущая проблема: при полном прогоне пайплайна результат нестабильный, и непонятно на каком уровне проблема — в тексте, вёрстке, фото или обложке. Всё рендерится одним проходом, и ошибки смешиваются.

Решение: разбить приёмку на 6 последовательных ворот. Каждые ворота добавляют один слой. Следующие ворота запускаются только после одобрения предыдущих Дашей.

## Протокол ворот

### Ворота 1 — Текст
**Что:** Stage3 / Phase B → финальный текст
**Формат выхода:** `.txt` + `.json` + отчёт агентов
**Что проверяем:** объём, нарратив, факты, структура глав, нет галлюцинаций
**Одобрение:** Даша читает `.txt`, даёт OK / правки
**Команда:** обычный Stage1→Stage2→Stage3→Phase B

### Ворота 2a — Чистый текст в PDF
**Что:** Stage4 с флагом `--text-only`
**Формат выхода:** PDF A5 — только абзацы текста, без декоративных элементов
**Что проверяем:** разбивка по страницам, заполненность (≥90%), шрифты, поля, нумерация
**Одобрение:** Даша смотрит PDF, даёт OK / правки рендерера
**Команда:** `python test_stage4.py --book $BOOK --text-only`

### Ворота 2b — Текст + bio_data
**Что:** Stage4 с флагом `--text-only --with-bio-block`
**Формат выхода:** PDF A5 — текст + ch_01 (хронология жизни, таблица данных, timeline)
**Что проверяем:** рендеринг bio_data блока, паспортные данные, хронология, корректность дат
**Одобрение:** Даша смотрит ch_01 в PDF, даёт OK / правки
**Команда:** `python test_stage4.py --book $BOOK --text-only --with-bio-block`

### Ворота 2c — Текст + bio_data + callout-ы/справки
**Что:** Stage4 с флагом `--no-photos`
**Формат выхода:** PDF A5 — текст + bio_data + callout-ы (цитаты) + исторические справки + плейсхолдеры для фото на chapter_start
**Что проверяем:** расположение цитат (только ch_03/04), исторических справок (только ch_02), плейсхолдеры на chapter_start, не ломают ли декоративные блоки вёрстку
**Одобрение:** Даша смотрит PDF, даёт OK / правки
**Команда:** `python test_stage4.py --book $BOOK --no-photos`

### Ворота 3 — Фото
**Что:** Stage4 с `--photos-dir`
**Формат выхода:** PDF A5 — полная книга с реальными фото (chapter_start + фотораздел)
**Что проверяем:** подбор фото для chapter_start, подписи, фотораздел (вертикальные по 1, горизонтальные по 2), нет микса ориентаций на одной странице
**Одобрение:** Даша смотрит PDF, даёт OK / заменить фото
**Команда:** `python test_stage4.py --book $BOOK --photos-dir $PHOTOS`

### Ворота 4 — Обложка
**Что:** Cover Designer отдельно
**Формат выхода:** AI-портрет + композиция обложки + финальный PDF
**Что проверяем:** выбор референс-фото (portrait first), качество AI-портрета, текст обложки (имя, годы — «род. ГГГГ» при death_year=null), цветовая схема
**Одобрение:** Даша смотрит обложку, даёт OK / новый референс / другой стиль
**Команда:** `python test_stage4.py --book $BOOK --photos-dir $PHOTOS --with-cover`

## Что нужно реализовать

### В pdf_renderer.py — флаги рендеринга

Добавить три флага, которые управляют какие слои рисовать:

| Флаг | Что рисуется | Что НЕ рисуется |
|------|-------------|-----------------|
| `--text-only` | Абзацы текста, заголовки глав, нумерация страниц | bio_data, callout-ы, ист.справки, фото, обложка |
| `--text-only --with-bio-block` | + bio_data блок ch_01 (timeline, таблица данных) | callout-ы, ист.справки, фото, обложка |
| `--no-photos` | + callout-ы, ист.справки, плейсхолдеры для фото | реальные фото, обложка |
| (без флагов) | Всё | — |

**Важно:** Layout Designer вызывается ОДИН РАЗ и генерирует полный layout JSON. Рендерер фильтрует слои при отрисовке. LLM-агенты не перезапускаются между воротами.

### Плейсхолдеры для фото (на воротах 2c и --no-photos)

На страницах chapter_start (ch_02, ch_03, ch_04) вместо реального фото рисуется:
- Серая рамка того же размера, что займёт фото
- Подпись: «[ФОТО — начало главы]»
- Это позволяет увидеть как фото повлияет на разбивку страниц

### В test_stage4.py — поддержка режимов

Пробрасывать флаги из командной строки в pdf_renderer:
- `--text-only` → рендерер рисует только текст
- `--with-bio-block` → добавляет bio_data
- `--no-photos` → добавляет callout-ы/справки + плейсхолдеры
- `--photos-dir` → реальные фото
- `--with-cover` → обложка с AI-портретом

### Чекпойнты между воротами

После каждых ворот сохранять чекпойнт:
- Ворота 1: `checkpoints/{project}/text_approved.json`
- Ворота 2a: `checkpoints/{project}/layout_text_approved.json`
- Ворота 2b: `checkpoints/{project}/layout_bio_approved.json`
- Ворота 2c: `checkpoints/{project}/layout_full_approved.json`
- Ворота 3: `checkpoints/{project}/photos_approved.json`
- Ворота 4: `checkpoints/{project}/cover_approved.json`

Следующие ворота не запускаются если предыдущий чекпойнт не одобрен.

## Пояснение архитектуры

Почему не нужны отдельные промпты для каждых ворот:

1. **Промпты остаются одни и те же** — Арт-директор и Верстальщик всегда генерируют полный layout JSON со всеми слоями (текст, callout-ы, фото-плейсхолдеры, bio_data)

2. **Фильтрация на уровне рендерера** — pdf_renderer.py при отрисовке смотрит на флаги и пропускает элементы, которые не нужны на текущих воротах. Например, при `--text-only` он видит callout в layout JSON, но не рисует его

3. **LLM вызывается один раз** — Layout Designer генерирует layout на воротах 2a, и этот же layout используется на 2b, 2c, 3, 4. Нет повторных вызовов API = нет дополнительных расходов

4. **Быстрая итерация** — если на воротах 2c видим что callout криво расположен, не нужно перегонять весь пайплайн. Правим рендерер → перерисовываем PDF из того же layout JSON

## Чеклист проверки

- `--text-only` рендерит PDF без callout-ов, ист.справок, фото, обложки
- `--text-only --with-bio-block` добавляет bio_data блок ch_01
- `--no-photos` добавляет callout-ы, ист.справки, плейсхолдеры
- Плейсхолдеры на chapter_start показывают размер будущего фото
- Полный запуск (без флагов) рендерит всё
- Чекпойнты сохраняются после каждых ворот
- Тест на Каракулиной: прогнать все 6 ворот последовательно
