Слайд 1 / 4 — Четыре сущности

Что настраивает Лена в админке. Каждая — отдельная вкладка.

🌐
Площадки
smm_platforms

Каналы публикации: Дзен, ВК, Telegram и т.д. У каждой площадки — промпт с описанием ЦА и общих требований.

Вложенные форматы: внутри каждой площадки — список форматов (статья, карточка с текстом, сторис). У каждого формата: название, slug, описание (длина, структура, визуальные требования).
Пример: Дзен → Статья (1500–3000 знаков, подзаголовки, обложка) · Карточка (изображение + 300 знаков)
📂
Рубрики
smm_rubrics

Тематические направления: о чём пишем. Промпт описывает содержание, тональность, источники.

Редактируется название: можно менять заголовок рубрики (не только промпт). Есть порядок сортировки, вкл/выкл.
Примеры: Советы по архивам · Истории клиентов · Бэкстейдж · Знаменитости
✍️
Журналисты
smm_journalists NEW

Три автора вместо одного. У каждого — свой промпт (стиль, тон, лицо повествования).

Назначения (галочки): под промптом — чекбоксы, какие рубрики и какие форматы журналист обслуживает. Не в тексте промпта — через UI.
Н Никита — советы, how-to, техническое, мужское лицо
Д Даша — клиентский сервис, истории клиентов, от лица сотрудника
Л Лена — общие темы, женское лицо, совладелец
🎭
Служебные роли
prompts (существующие)

Остаются как есть: Стратегия, Скаут, Редактор, Иллюстратор. Промпт каждой роли — одна текстовая область.

Что убирается: единственный «Журналист» из ролей — его заменяют три персонажа выше.
Стратегия · Скаут · Редактор (выпускающий) · Иллюстратор

Как система выбирает журналиста
① Скаут генерирует тему
Тема приходит с rubric_slug (рубрика) + format_slug (формат площадки)
② Поиск журналиста
Система ищет журналиста, у которого оба чекбокса стоят: и эта рубрика, и этот формат. Если несколько — первый по приоритету.
③ Генерация текста
system = промпт выбранного журналиста. user = тема + промпт рубрики + описание формата.

Слайд 3 / 4 — Пайплайн: от идеи до публикации

Полная цепочка и что на вход получает каждый агент.

1
🔍 Скаут — генерирует контент-план
Получает (промпты):
✦ system: Скаут (роль)
✦ context: Стратегия (роль)
✦ context: Площадка (промпт выбранной площадки)
✦ context: все активные Рубрики (промпты)
✦ context: список Форматов этой площадки
✦ context: отзывы клиентов из БД
✦ context: идеи от Лены (если введены)
Отдаёт:
JSON-массив тем. Каждая тема содержит: topic, angle, rubric_slug, format_slug
Скаут теперь указывает и рубрику, и формат для каждой темы.
2
✍️ Журналист (автоматически выбранный) — пишет текст
Получает (промпты):
✦ system: промпт конкретного журналиста (стиль, тон, лицо)
✦ user: тема
✦ user: промпт рубрики (содержание, источники)
✦ user: описание формата (длина, структура)
Отдаёт:
Готовый текст: заголовок + тело статьи
Изменение vs. v1: журналист теперь получает промпт рубрики + описание формата. Раньше получал только тему.
3
🎨 Иллюстратор — визуальный концепт + картинка
Получает (промпты):
✦ system: Иллюстратор (роль)
✦ user: заголовок + фрагмент текста статьи
Отдаёт:
Image prompt → Replicate → файл обложки
Без изменений vs. v1
4
📋 Выпускающий редактор — решение о публикации
Получает (промпты):
✦ system: Редактор (роль)
✦ user: заголовок + полный текст + image prompt
Отдаёт:
{approved, comment} → статус «ready» или «editor_rejected»
Без изменений vs. v1
5
👁 Лена — одобряет, ставит дату, публикует
Новые действия в v2:
✦ Назначить дату публикации (date-picker)
✦ Удалить пост с любой стадии
✦ Сортировка/фильтрация по дате
Далее:
Одобренный пост с датой → публикация на площадке (Playwright / API)

Слайд 4 / 4 — UI контент-плана и БД

Новые поля, удаление, даты, схема таблиц.

Изменения в интерфейсе
📅 Дата публикации
Новая колонка publish_date на карточке поста. Лена ставит / снимает дату через date-picker. Посты сортируются по дате (ближайшие сверху).
🗑 Удаление постов
Кнопка удаления на каждой карточке, на любой стадии. Подтверждение через модалку. Мягкое удаление (статус → deleted) или жёсткое — обсудим.
🏷 Бейджи на карточках
Каждый пост показывает: рубрику, формат, журналиста (кто написал), площадку. Цвет бейджа — по журналисту.
✍️ Настройки → Журналисты (новая вкладка)
Вместо одной роли «Журналист» — вкладка со списком авторов. Для каждого: имя, промпт, чекбоксы рубрик и форматов. Удаление журналиста автоматически снимает все назначения.
Новые таблицы / поля в БД
-- Новая таблица: форматы площадок
CREATE TABLE
smm_formats (
id          SERIAL PRIMARY KEY,
platform_id  INTEGER REFERENCES smm_platforms(id),
slug         TEXT NOT NULL,
name         TEXT NOT NULL,
description  TEXT DEFAULT '',
is_active    BOOLEAN DEFAULT TRUE
);
-- Новая таблица: журналисты
CREATE TABLE
smm_journalists (
id          SERIAL PRIMARY KEY,
slug         TEXT UNIQUE NOT NULL,
name         TEXT NOT NULL,
prompt_key   TEXT NOT NULL,
is_active    BOOLEAN DEFAULT TRUE
);
-- Назначения: журналист ↔ рубрики + форматы
CREATE TABLE
smm_journalist_assignments (
journalist_id INTEGER REFERENCES smm_journalists(id) ON DELETE CASCADE,
rubric_id     INTEGER REFERENCES smm_rubrics(id) ON DELETE CASCADE,
format_id     INTEGER REFERENCES smm_formats(id) ON DELETE CASCADE
);
-- Новые поля в smm_posts
ALTER TABLE
smm_posts ADD
journalist_id  INTEGER REFERENCES smm_journalists(id),
format_id      INTEGER REFERENCES smm_formats(id),
rubric_id      INTEGER REFERENCES smm_rubrics(id),
publish_date   DATE
;
Что убирается
  • ✕ Роль smm_journalist из SMM_ROLES — заменяется таблицей smm_journalists
  • ✕ Понятие «один журналист на все рубрики»
Что не меняется
  • ✓ Таблицы smm_platforms, smm_rubrics — остаются, расширяются
  • ✓ Роли: Стратегия, Скаут, Редактор, Иллюстратор — без изменений
  • ✓ Общий пайплайн: Скаут → Журналист → Иллюстратор → Редактор
Вопросы для обсуждения с Леной
1. Удаление постов — мягкое (статус «deleted», можно восстановить) или жёсткое (DELETE из БД)?
2. Дата публикации — ставится при одобрении, или на любой стадии? Пока: при одобрении. Позже — контент-план от маркетолога.
3. Если журналист не найден (нет подходящего по рубрике+формату) — fallback на дефолтный промпт, или не генерировать?
4. Форматы пока описываются текстом внутри промпта площадки, или уже отдельными полями (slug, name, description)? Рекомендация: отдельные поля → чище и удобнее для назначений.
