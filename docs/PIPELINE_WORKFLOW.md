# Как работать с пайплайном GLAVA

Это руководство для **Даши** (промпт-дизайнер) и **разработки** (инженеры).  
Описывает: как устроен пайплайн, как вносить изменения, как тестировать и деплоить.

---

## Общая картина

```
Голосовые / фото
      ↓
[Stage 1] Транскрипция → Чистильщик → Экстрактор фактов
      ↓ fact_map.json + cleaned_transcript.txt
[Stage 2] Историк → Гострайтер → Фактчекер
      ↓ book_draft.json
[Stage 3] Литературный редактор → Корректор
      ↓ proofreader_report.json  ← финальный текст
[Stage 4] Обложка → Арт-директор → Верстальщик → QA
              ↕                              ↕
        [Интервьюер]                  [Сборка PDF]
              ↓                              ↓
      interview_questions.json     karakulina_book.pdf
```

Каждый этап — это:
- **Промпт** (в админке и в `prompts/`) — что агент умеет делать
- **Карточка** (в `docs/stages/`) — зачем этот агент и как с ним работать
- **Конфиг** (`prompts/pipeline_config.json`) — модель, токены, температура
- **Код** (`pipeline_utils.py`, `scripts/test_stageN_*.py`) — как агент вызывается

## Роли

| Кто | Что делает |
|-----|------------|
| **Даша** | Пишет и улучшает промпты; проверяет качество вывода агентов; обновляет секцию «Для Даши» в карточках |
| **Разработка** | Пишет оркестраторы, передаёт правильные данные агентам; обновляет секцию «Для разработки» в карточках; деплоит изменения |
| **Оба** | Фиксируют проблемы в карточках этапов; согласуют изменения через PR или чат |

## Где что хранится

```
prompts/
  pipeline_config.json        ← единый конфиг: модели, токены, ссылки на файлы промптов
  01_cleaner_v1.md            ← промпты (версионируются суффиксом _v1, _v2, ...)
  02_fact_extractor_v3.2.md
  ...

docs/stages/
  STAGE_TEMPLATE.md           ← шаблон карточки (этот файл)
  PIPELINE_WORKFLOW.md        ← этот гайд
  01_cleaner.md               ← карточки этапов (одна на агента)
  02_fact_extractor.md
  ...

scripts/
  test_stage1_*.py            ← тестовые прогоны по проекту
  test_stage2_*.py
  test_stage4_*.py
  build_*_pdf.py              ← сборщики PDF

exports/
  project_stage_agent_ts.json ← все артефакты прогонов
```

## Как вносить изменения в промпт

### Шаг 1 — Диагностика
Прежде чем менять промпт, зафиксируй проблему:
- Что именно не так в выводе? (конкретный пример)
- Это системная ошибка или разовая?
- В какой части промпта скорее всего причина?

Запиши в таблицу «Типичные ошибки» в карточке этапа.

### Шаг 2 — Изменение
1. **Создай новую версию файла** — не перезаписывай старый:
   ```
   02_fact_extractor_v3.2.md → 02_fact_extractor_v3.3.md
   ```
2. Внеси изменения.
3. Обнови карточку этапа (`docs/stages/02_fact_extractor.md`):
   - Добавь строку в «Историю изменений»
   - Обнови «Типичные ошибки» если нужно

### Шаг 3 — Тест локально (Даша)
Запроси у разработки тестовый прогон на реальном проекте:
```
"Даша, можешь прогнать v3.3 на Каракулиной? Хочу проверить что gaps стали корректнее."
```

### Шаг 4 — Тест (разработка)
```bash
# Переключить версию в конфиге:
# prompts/pipeline_config.json → "prompt_file": "02_fact_extractor_v3.3.md"

python scripts/test_stage1_karakulina.py
# Проверить: exports/karakulina_fact_map_*.json
```

### Шаг 5 — Деплой
Разработка обновляет `pipeline_config.json` на сервере:
```bash
scp prompts/pipeline_config.json glava:/opt/glava/prompts/
```

**Старая версия промпта остаётся на сервере** — можно откатиться за 1 минуту.

### Шаг 6 — Откат (если нужен)
```json
// pipeline_config.json: просто вернуть старое имя файла
"prompt_file": "02_fact_extractor_v3.2.md"
```

## Как добавить новый этап

1. **Разработка:** добавляет вызов агента в оркестратор
2. **Даша:** пишет промпт `XX_new_agent_v1.md`
3. **Оба:** заполняют карточку `docs/stages/XX_new_agent.md` по шаблону
4. **Разработка:** добавляет секцию в `pipeline_config.json`
5. Тест → деплой по схеме выше

## Как тестировать отдельный этап

Каждый этап можно запустить изолированно, подав ему готовые входные данные:

```bash
# Stage 1 (Чистильщик + Факт-экстрактор)
python scripts/test_stage1_karakulina.py
# → exports/karakulina_cleaned_transcript.txt
# → exports/karakulina_fact_map_*.json

# Stage 2 (Гострайтер)
python scripts/test_stage2_karakulina.py
# → exports/karakulina_book_*.json

# Stage 4 (Обложка + Вёрстка)
python scripts/test_stage4_karakulina.py --photos-dir exports/karakulina_photos/
# → exports/karakulina_stage4_cover_portrait_*.webp
# → exports/karakulina_stage4_layout_*.json

# Сборка PDF из данных пайплайна
python scripts/build_karakulina_pdf.py
# → exports/karakulina_book_STAGE4_*.pdf
```

## Версионирование промптов

| Что изменилось | Версия |
|----------------|--------|
| Мелкая правка формулировки | v3.1 → v3.2 |
| Новое правило или структурное изменение | v3.x → v4.0 |
| Переписан с нуля / новая логика | v4.x → v5.0 |

Старые версии **не удаляются** — они нужны для отката и сравнения.  
Активная версия всегда в `pipeline_config.json`.

## Что хранить в админке

Админка (`cabinet/app.py`) — рабочее место Даши.  
В ней удобно хранить:

| Объект | Где |
|--------|-----|
| Текст активного промпта | Админка (редактируемый) |
| Ссылка на карточку этапа | Админка (читаемый, ссылка на `docs/stages/`) |
| История версий промпта | `pipeline_config.json` → `_notes` |
| Тест-кейсы и ожидаемый вывод | Карточка этапа + `docs/TESTING.md` |

**Принцип:** промпт редактируется в админке → сохраняется на сервер как новая версия → разработка подтягивает в `pipeline_config.json`.

## Чек-лист перед деплоем нового промпта

- [ ] Промпт протестирован на минимум одном реальном проекте
- [ ] Результат проверен Дашей вручную
- [ ] Карточка этапа обновлена (версия, дата, что изменилось)
- [ ] Старая версия файла сохранена (не перезаписана)
- [ ] `pipeline_config.json` обновлён с новым именем файла и записью в `_notes`
- [ ] Разработка уведомлена если изменился формат входных/выходных данных

## FAQ

**Q: Можно ли менять промпт напрямую на сервере?**  
A: Технически да, но нежелательно — тогда git-история и локальные файлы расходятся. Лучше менять локально и деплоить через scp.

**Q: Что если агент вернул невалидный JSON?**  
A: Смотри логи (`/tmp/stage*.log`). Обычно причина — агент превысил `max_tokens` или запутался в структуре. Решение: поднять `max_tokens` в конфиге или упростить промпт.

**Q: Как понять что температура правильная?**  
A: Для структурированных выводов (JSON, факты, код) — `0.0–0.2`. Для текста (биография, вопросы) — `0.4–0.7`. Высокая температура у фактчекера — баг, у гострайтера — норма.

**Q: n8n нужен?**  
A: Нет. Пайплайн реализован в Python, этого достаточно на текущем масштабе. n8n добавил бы ещё один сервис без пропорциональной пользы.
