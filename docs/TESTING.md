# Система тестирования бота GLAVA

Документ задаёт общую логику тестирования, периодичность прогонов, формат отчётов и порядок реагирования на результаты. Тест-кейсы взяты из «Тест-кейсы для бота» (USER_SCENARIOS).

---

## 1. Общая логика

### 1.1 Что тестируем

- **Поведение бота по сценариям пользователя**: навигация до оплаты, конфигурация заказа (персонажи + email), оплата, блокировка контента до оплаты, восстановление черновика, команды `/list` и `/cabinet`.
- **Регрессии**: изменения в коде не должны ломать уже работающие сценарии.
- **Предусловия**: тесты предполагают пользователя U1 — новый аккаунт (нет draft, нет project). Цена: 990 ₽/персонаж.

### 1.2 Источник тест-кейсов

Используется набор **TC-01 … TC-27** из документа «Тест-кейсы для бота»:

| Блок | Тест-кейсы | Содержание |
|------|------------|------------|
| **A. Pre-pay навигация** | TC-01 … TC-03 | /start, Пример, Стоимость, Назад |
| **B. Pre-pay конфигурация** | TC-04 … TC-13 | Персонажи (ввод, добавление, удаление), email, валидация, Итого |
| **C. Оплата** | TC-14 … TC-17 | Перейти к оплате, payment_pending, Проверить оплату |
| **D. Блокировка до оплаты** | TC-18 … TC-21 | Голос, фото, аудио/документ, произвольный текст |
| **E. Resume (draft)** | TC-22 … TC-24 | /start при разных состояниях черновика |
| **F. /list и /cabinet** | TC-25 … TC-27 | /list до оплаты, /cabinet (пароль OK / короткий) |

### 1.3 Способ прогона

- **Автотесты**: все TC-01 … TC-27 закрыты unit-тестами в `tests/test_bot_flows.py` (моки БД и Telegram API).
- **Ручной прогон**: при необходимости дублировать проверку в реальном боте — по чек-листу в Telegram.

## 2. Протокол запуска тестов (явки, пароли, окружение)

### 2.1 Требования к окружению

- **Python**: 3.10+ (рекомендуется 3.11+).
- **Зависимости для автотестов**: `pytest`, `pytest-asyncio`. Остальные зависимости проекта (telegram, db, storage и т.д.) подставляются из `requirements.txt` при обычной разработке.

Установка только для тестов:

```bash
pip install pytest pytest-asyncio
```

Либо из корня проекта (уже включено в полный набор):

```bash
pip install -r requirements.txt
```

### 2.2 Пароли и ключи

- **Автотесты не используют реальные ключи и пароли.**  
  В тестах замоканы: `db`, `db_draft`, `storage`. Ни `BOT_TOKEN`, ни `YANDEX_API_KEY`, ни `ASSEMBLYAI_API_KEY`, ни доступ к БД/облаку не требуются. Запуск безопасен на любой машине без .env и секретов.

- **Ручной прогон** (если решите дублировать в Telegram): нужен тестовый бот (отдельный токен) и при необходимости тестовая БД/оплата. Реальные прод-ключи для ручного прогона не использовать.

### 2.3 Команды запуска

Из **корня проекта** (каталог с `main.py`, `tests/`, `scripts/`):

**Только тесты (консоль):**

```bash
python -m pytest tests/test_bot_flows.py -v --tb=short
```

**Тесты + отчёт (MD и PDF):**

```bash
python scripts/run_test_report.py
```

Скрипт сам запускает pytest, собирает результаты по TC-01 … TC-27 и кладёт отчёт в `docs/test-runs/test-run_YYYY-MM-DD_HHmm.md` и `.pdf`.

### 2.4 Где лежат отчёты

- Каталог: `docs/test-runs/`.
- Имена: `test-run_YYYY-MM-DD_HHmm.md`, `test-run_YYYY-MM-DD_HHmm.pdf`.
- Шаблон для ручного заполнения: `docs/test-runs/REPORT_TEMPLATE.md`.

### 2.5 Кто запускает (явки)

- **Перед релизом**: ответственный за релиз или разработчик обязан прогнать автотесты и при необходимости ручной прогон; отчёт сохранить в `docs/test-runs/`.
- **Регулярный прогон**: по договорённости (например, раз в 1–2 недели) — любой участник команды с доступом к репозиторию может запустить `python scripts/run_test_report.py` и выложить/прислать отчёт.
- Для автотестов **доступ в прод, пароли и ключи не нужны** — достаточно клона репозитория и установленных зависимостей.

## 3. Периодичность прогонов

| Событие | Когда запускать | Объём |
|--------|------------------|--------|
| **Перед релизом** | Перед выкладкой изменений на прод | Полный прогон TC-01 … TC-27 |
| **После правок в сценариях** | После изменений в хендлерах заказа, оплаты, резюме, /cabinet | Блоки A–F, затронутые правками |
| **Регулярная проверка** | Раз в 1–2 недели (по возможности) | Полный прогон или критичные блоки (A, B, C, D) |
| **По запросу** | После багфиксов или по запросу команды | Выборочные кейсы или полный прогон |

Критичные для бизнеса блоки: **A** (первый контакт), **B** (оформление), **C** (оплата), **D** (блокировка до оплаты). Их желательно не пропускать перед релизом.

## 4. Формат отчёта

Отчёт по прогону — один файл/таблица на прогон.

### 4.1 Обязательные поля

- **Дата и время прогона**
- **Версия/коммит** (если применимо): ветка или хэш
- **Окружение**: тестовый бот / прод (если вдруг прогон по проду)
- **Исполнитель**: кто прогонял

### 4.2 Таблица результатов

Для каждого тест-кейса:

| ID | Название | Результат | Примечание |
|----|----------|-----------|------------|
| TC-01 | /start → главное меню | ✅ Pass / ❌ Fail | |
| TC-02 | Пример → Назад | ✅ / ❌ | |
| … | … | … | |

- **Результат**: **✅ Pass** или **❌ Fail**.
- **Примечание**: кратко — что именно не так (текст ответа бота, отсутствие кнопки, неверный экран и т.п.).

### 4.3 Итог по прогону

- **Всего**: X из 27 (или актуальное число кейсов).
- **Pass**: N.
- **Fail**: M.
- **Критичные падения** (если есть): список TC-XX и кратко суть.

Пример сводки:

```
Прогон: 2026-02-03 14:00
Ветка: main, коммит abc123
Окружение: тестовый бот
Исполнитель: Иванов

Итог: 25/27 Pass, 2 Fail.
Fail: TC-12 (неверный email), TC-19 (фото до оплаты).
```

### 4.4 Где хранить отчёты

- **Рекомендация**: папка `docs/test-runs/` в репозитории.
- Имя файла: `test-run_YYYY-MM-DD_HHmm.md` и `test-run_YYYY-MM-DD_HHmm.pdf`.
- Шаблон отчёта: `docs/test-runs/REPORT_TEMPLATE.md`.
- **Автогенерация отчёта (MD + PDF):** из корня проекта выполнить  
  `python scripts/run_test_report.py`  
  Скрипт при наличии pytest запускает автотесты и подставляет результаты в отчёт; иначе формируется шаблон для ручного прогона.

## 4. Реагирование на результаты

### 4.1 Все кейсы прошли (Pass)

- Релиз разрешён (если прогон был предрелизный).
- Отчёт сохраняется для истории.

### 5.2 Есть падения (Fail)

1. **Зафиксировать**: в отчёте указать ID кейса, шаги, фактическое поведение, при необходимости — скрин/лог.
2. **Оценить критичность**:
   - **Критично**: оплата (C), блокировка до оплаты (D), невозможность оформить заказ (B). Без фикса или обходного решения релиз не делать.
   - **Некритично**: косметика, редкий сценарий. Решение — фикс в текущем или следующем спринте; при необходимости — явно описать в релизных заметках.
3. **Завести задачу**: баг или доработка в трекере (если есть) с ссылкой на тест-кейс и отчёт.
4. **Повторный прогон**: после фикса — прогнать затронутые кейсы (минимум) или полный набор (перед следующим релизом).

### 4.3 Регулярный прогон выявил регрессию

- То же, что п. 4.2: фиксация, оценка критичности, задача, повторный прогон после фикса.
- Имеет смысл закрепить в процессе: «регрессия по тест-кейсам = приоритетный фикс».

### 5.4 Невозможно прогнать (бот недоступен, падает окружение)

- В отчёте указать: «Прогон не выполнен: причина».
- Релиз без прогона — только по явному решению (и с пометкой в релизных заметках).
