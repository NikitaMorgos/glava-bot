# Спецификация модуля «Финансы» (GLAVA) — для переноса по аналогии

Документ описывает, как в проекте GLAVA устроен раздел финансов в админке: ручные расходы, учёт доходов (в т.ч. с заделом под платёжку), сводный P&L. Его можно передать разработчику другого проекта как ориентир по данным, экранам и поведению.

## Исходники в этом репозитории

| Назначение | Путь |
|------------|------|
| HTTP-маршруты (Flask blueprint) | `admin/blueprints/finance.py` |
| SQL и бизнес-логика БД | `admin/db_finance.py` |
| Регистрация blueprint | `admin/app.py` (импорт `finance_bp`, `register_blueprint`) |
| DDL и сиды | `scripts/migrate_finance.py` |
| Шаблоны UI | `admin/templates/finance/*.html` |

## Доступ и безопасность

- Раздел доступен только авторизованным пользователям с ролью из множества: `dev`, `dasha`, `lena` (см. `ALL_ROLES` в `finance.py`).
- Иначе — редирект на страницу логина.
- При записи расходов и ручном добавлении дохода в поле `created_by` кладётся `session.get("username", "")`.

## URL-структура (префикс `/finance`)

| Путь | Назначение |
|------|------------|
| `/finance/` | Редирект на список расходов |
| `/finance/expenses` | Расходы; query `month=YYYY-MM` — фильтр по месяцу |
| `/finance/expenses/add` | POST — новый расход |
| `/finance/expenses/<id>/edit` | GET/POST — редактирование |
| `/finance/expenses/<id>/delete` | POST — удаление |
| `/finance/categories/add`, `/categories/<id>/delete` | POST — добавить / удалить категорию расходов |
| `/finance/initiators/add`, `/initiators/<id>/delete` | POST — добавить / удалить «инициатора» |
| `/finance/income` | Доходы; query `month=YYYY-MM` |
| `/finance/income/add` | POST — новое поступление |
| `/finance/income/<id>/edit`, `/income/<id>/delete` | GET/POST и POST — правка / удаление |
| `/finance/pnl` | P&L; query `months=N` (по умолчанию 6) |

## Модель данных (PostgreSQL)

### Расходы

- **`expense_categories`**: `id`, `name` (unique), `sort_order`, `created_at`.
- **`expense_initiators`**: `id`, `name` (unique), `created_at` — кто инициировал трату (роль/человек и т.д.).
- **`expenses`**:
  - `date` (DATE), `amount` (NUMERIC(12,2))
  - `category_id` → `expense_categories`, `initiator_id` → `expense_initiators`
  - `periodicity`: `разовая` \| `подписка`
  - `behavior`: `постоянная` \| `переменная`
  - `title`, `comment` (опционально)
  - `created_at`, `created_by`

Удаление категории и инициатора разрешено только если нет связанных строк в `expenses`.

### Доходы (поступления)

- **`finance_income`**:
  - `date`, `amount`
  - `title`, `comment` (опционально)
  - `source` — текст, по умолчанию в схеме ориентир на платёжку, например `ЮKassa`
  - `created_at`, `created_by`

### P&L

Логика в `get_pnl()` в `db_finance.py`:

- Берётся до **N** последних календарных месяцев, в которых есть **хотя бы один** расход **или** доход (объединение `DISTINCT` по `YYYY-MM` из обеих таблиц).
- Расходы агрегируются **по названию категории** и месяцу.
- Доходы для отчёта — **одна сводная строка выручки по месяцам** (сумма по `finance_income`), без детализации отдельных платежей внутри таблицы P&L.
- Итог по месяцу: выручка минус сумма расходов.

## Поведение UI (ожидания)

- Суммы вводятся как десятичные; запятая в форме может заменяться на точку перед парсингом.
- Сумма должна быть **строго положительной**; дата обязательна.
- После успешного добавления/правки часто редирект обратно с `month=YYYY-MM` из даты операции, чтобы пользователь остался в том же месяце.
- Flash-сообщения об ошибках валидации и успехе операций.

## Ручные расходы vs доходы

- **Расходы** в GLAVA заводятся **вручную** через формы (категория, инициатор, периодичность, поведение, комментарий).
- **Доходы** в таблицу `finance_income` попадают через **те же CRUD-формы** в админке; вызов вставки — функция `add_income()` в `db_finance.py`.

## Автоматические доходы (как довести до конца)

В текущем коде GLAVA **нет** отдельного обработчика webhook/cron, который пишет в `finance_income` при оплате. Модель это допускает: достаточно при успешной оплате вызывать ту же операцию, что и при ручном вводе (вставка в `finance_income`), передавая осмысленные `source`, `title`/`comment`, дату и сумму.

Рекомендации для продакшена при автозаписи:

- **Идемпотентность**: уникальный ключ внешнего id платежа (отдельная колонка или таблица связей), чтобы повтор webhook не дублировал строку.
- **`created_by`**: например строка `system` или имя интеграции.

## Миграция и сиды

- Запуск: `python scripts/migrate_finance.py` (требуется `DATABASE_URL` в окружении).
- Скрипт создаёт таблицы и подмешивает начальные категории и инициаторов (см. константы в файле).

## Что перенести «по аналогии» на другой стек

Минимальный набор для эквивалентного поведения:

1. Те же сущности БД (или упрощение: без периодичности/поведения, если не нужны).
2. Справочники категорий/инициаторов с защитой от удаления при наличии ссылок.
3. CRUD расходов и доходов с фильтром по месяцу.
4. Отчёт P&L: расходы по категориям × месяцы + одна строка выручки × месяцы + прибыль/убыток.
5. Тот же слой записи доходов для ручного ввода и для будущей автоматизации платежей.
