## [2026-04-27] — Completeness Auditor v1.0 (задача 014)

**Файл:** `16_completeness_auditor_v1.md`
**Версия:** v1.0
**Изменил:** Cursor (task 014)

### Что добавлено
- Новый агент 16 — Completeness Auditor, работает после Fact Extractor
- Модель: `claude-haiku-4-5-20251001` (паттерн-матчинг, Sonnet не нужен)
- Входные данные: cleaned_transcript + fact_map от FE
- Двухсекционный вывод: `auto_enrich` (high-confidence → автоматически в fact_map) +
  `log_only_gaps` (low-confidence → только warning в roles_checklist и manifest)
- Семантический критерий значимости: добавлять только при наличии родства/роли/участия в событии
- Blocklist исторических фигур (детерминированная защита от false positives)
- Правило рассказчика как персонажа биографии (Task 011 / 014: Татьяна должна быть в persons[])
- `source_quote` обязателен для каждого `auto_enrich`-элемента

### Сопутствующие изменения (не промпты)
- `pipeline_config.json`: добавлена секция `completeness_auditor`
- `pipeline_utils.py`: `run_completeness_auditor()`, `apply_completeness_enrichment()`
- `scripts/normalize_named_entities.py`: Name Normalizer (детерминированный, без LLM)
- `scripts/compare_persons_across_runs.py`: регрессионное сравнение persons[] между прогонами
- `scripts/test_stage1_karakulina_full.py`: интеграция CA + Name Normalizer после FE

### Зачем
Системная нестабильность LLM Fact Extractor: в разных прогонах исчезают персоны и факты.
Правкой промпта FE не лечится. CA + Name Normalizer — постпроцессинговая защита.

### Ожидаемый результат
Stability score persons[] ≥ 90% между двумя прогонами на одних транскриптах.
