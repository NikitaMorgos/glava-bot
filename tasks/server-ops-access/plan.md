# Задача: Автономный операционный доступ агента к серверу и n8n

## Цель

Агент (Cursor AI) получает самостоятельный доступ к серверу и n8n admin panel, чтобы без участия пользователя:
- Диагностировать ошибки и инциденты по логам
- Вносить фиксы в код и деплоить
- Проверять критический happy path
- Эскалировать только при: внешний провайдер сломан / нет доступов / риск потери данных

## Логика действий при инциденте

```
Ошибка/жалоба пользователя
  → 1. Агент сам заходит на сервер, смотрит логи (journalctl)
  → 2. Локализует поломку (бот / флask / n8n / БД / внешний API)
  → 3. Вносит фикс в код (локально, через Cursor)
  → 4. Деплоит: git push → git pull на сервере → restart services
  → 5. Проверяет health endpoint + critical path
  → 6. Эскалирует пользователю ТОЛЬКО если: внешний провайдер / нет доступа / риск данных
```

## Шаги реализации

### Шаг 1 — SSH ключ и конфиг (ТЕКУЩИЙ)
- Использовать существующий ключ `~/.ssh/id_ed25519`
- Создать `~/.ssh/config` с алиасом `glava`
- Добавить публичный ключ в `authorized_keys` на сервере
- Проверить: `ssh glava "echo ok"` без пароля

### Шаг 2 — N8N API Key
- Создать API key в n8n (Settings → API → Create)
- Добавить в `.env`: `N8N_API_KEY=...`
- Добавить в `.env` сервера
- Проверить: `curl -H "X-N8N-API-KEY: ..." http://localhost:5678/api/v1/workflows`

### Шаг 3 — ops.sh на сервере
- Обновить/создать `/opt/glava/ops.sh` с командами:
  - `logs-admin`, `logs-bot`, `status`, `health`
  - `deploy`, `seed-prompts`, `n8n-status`
  - `n8n-workflows` (список воркфлоу через API)
  - `n8n-executions` (последние execution'ы)

### Шаг 4 — Проверка автономного доступа
- Войти без пароля: `ssh glava "bash /opt/glava/ops.sh health"`
- Проверить n8n API через ops.sh
- Задокументировать схему доступа в breadcrumbs
